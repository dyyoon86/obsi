> 사내망에 깔리는 AI 풀스택 한 장 이해.
> 비유 + 평이한 설명 + "이게 왜 필요?" 위주.

## 0. 전체 그림 한눈에

```
┌─────────────────────────────────────────────────────────┐
│           [직원 — 개발자/QA/상담/운영]                    │
│                       ↓                                   │
│              [웹 UI / API / Slack]                       │
│                       ↓                                   │
│         ┌──────────  K3s (Kubernetes) ──────────┐        │
│         │    Helm 으로 깔린 AI 서비스 묶음        │        │
│         │                                       │        │
│         │  LLM(Qwen, Gemma) · 임베딩 · 리랭커    │        │
│         │  ASR · VL · 화자분리 · OCR · 정렬       │        │
│         │                                       │        │
│         │  + PostgreSQL/pgvector (검색 DB)       │        │
│         │  + Prometheus/Grafana (모니터링)       │        │
│         └────────────────────────────────────────┘        │
│                       ↓                                   │
│         ┌────────  NVIDIA GPU 8장 ─────────┐             │
│         │  CUDA / cuDNN / TensorRT / NCCL  │             │
│         │  GPU 드라이버 + Container Toolkit │             │
│         └───────────────────────────────────┘            │
│                       ↓                                   │
│              [Ubuntu 22.04 베어메탈]                     │
└─────────────────────────────────────────────────────────┘
```

비유:
- **GPU 스택** = 식당의 가스레인지 + 식기 (요리할 장비)
- **컨테이너 스택** = 주방 동선/룰 (누가 어디서 뭘 만드는지)
- **AI 모델** = 셰프들 (각자 한식/일식/제과 전문)
- **DB/모니터링** = 재고관리 + 매출판
- **Ubuntu** = 식당 건물 그 자체

## 1. 인프라 & 라이브러리

### 1-1. OS

#### Ubuntu 22.04 ✅

**무엇**: 리눅스 운영체제. 우분투 = 리눅스 배포판 중 가장 인기.
**왜 22.04?**: LTS(Long Term Support, 5년 보장). 안정성이 가장 중요.
**비유**: 식당 건물. 2027년까지 보안 패치 무료.

---

### 1-2. GPU 스택

> GPU 가 일하려면 **드라이버 → CUDA → cuDNN/NCCL → 컨테이너 연결 → vLLM** 순으로 쌓아야 함.

#### GPU 드라이버 580.142 ✅

**무엇**: GPU 하드웨어와 OS 가 대화하는 통역기.
**왜 필요**: 드라이버 없으면 OS 가 GPU 를 그냥 "그래픽카드 비스무리한 거" 정도로만 인식. AI 계산 못 함.
**비유**: 컴퓨터 ↔ GPU 사이의 통역사.

#### CUDA 13.0 ✅

**무엇**: NVIDIA 가 만든 **GPU 에서 일반 계산을 시키는 프로그래밍 도구**.
**왜 필요**: GPU 는 원래 그래픽용. CUDA 덕분에 AI/수치계산도 GPU 에서 가능.
**비유**: 가스레인지 자체가 GPU 라면, CUDA 는 "가스 밸브 + 사용법".
**왜 13.0?**: 최신 라인. 신규 모델/라이브러리들이 13.0 요구.

#### CUDA Toolkit 13.0 ✅

**무엇**: CUDA 로 개발할 때 쓰는 **컴파일러 + 라이브러리 모음**.
**왜 필요**: 개발자/모델이 "GPU 야 이것 좀 계산해" 라고 부탁할 수 있게 해줌.
**비유**: 요리 도구 풀세트.

#### cuDNN (CUDA 13.0 호환) ✅

**무엇**: **딥러닝 전용 가속 라이브러리**. CUDA 위에 얹는 거.
**왜 필요**: AI 모델의 핵심 연산 (행렬곱, convolution) 을 GPU 가 최대 속도로 처리하게 해줌.
**비유**: 일반 가스레인지에 추가한 "고온 강불 모드".

#### NCCL (CUDA 13.0 호환) ✅

**무엇**: NVIDIA Collective Communication Library. **여러 GPU 가 서로 데이터 주고받는 라이브러리**.
**왜 필요**: 큰 모델 (Qwen 35B) 은 GPU 한 장에 안 들어감 → 4장에 나눠 올림 → 4장이 빠르게 결과 합쳐야 함. NCCL 이 그 통신 담당.
**비유**: 4명의 셰프가 한 요리를 나눠 만들 때 쓰는 무전기.

#### NVIDIA Container Toolkit 1.19.1-1 ✅
#### nvidia-container-runtime 1.19.1-1 ✅
#### libnvidia-container 1.19.1-1 ✅

**무엇**: 컨테이너 (Docker/Podman/Kubernetes) 안에서 GPU 를 쓸 수 있게 해주는 다리.
**왜 필요**: 기본 컨테이너는 격리돼서 GPU 못 봄 → 이 toolkit 이 "이 컨테이너 GPU 써도 됨" 허락.
**비유**: 식당 직원에게 발급하는 가스레인지 사용 허가증.
**세 가지 차이**:
- Toolkit = 사용자가 호출하는 명령어
- Runtime = 실제로 GPU 연결하는 엔진
- libnvidia-container = 그 둘이 공유하는 핵심 라이브러리

→ 셋이 같은 버전이어야 호환됨.

#### NVIDIA TensorRT (CUDA 13.0 호환) ✅

**무엇**: AI 모델을 **GPU 에서 최대한 빠르게 돌도록 최적화**해주는 도구.
**왜 필요**: 같은 모델도 TensorRT 통과시키면 2~5배 빨라짐. 답변 지연 단축.
**비유**: 운전 잘하는 운전기사 → 같은 차로 더 빠르게.
**언제 씀**: 운영 단계에서 모델 deploy 할 때. 개발 중엔 안 씀.

#### vLLM 0.21.0 📋

**무엇**: **LLM 을 효율적으로 서빙하는 서버 프로그램**. UC Berkeley 발 오픈소스.
**왜 필요**: LLM 을 그냥 띄우면 한 번에 1명만 처리. vLLM 은 동시에 수십~수백 명 처리하면서도 메모리 절약.
**핵심 기술**: **PagedAttention** — GPU 메모리를 OS 페이징처럼 관리해서 낭비 없앰.
**비유**: 한 셰프가 동시에 10개 요리를 효율적으로 처리하는 시스템.
**📋 상태**: 아직 설치 대기 (모델 띄우기 직전 단계).

### 1-3. 컨테이너 스택

> 여러 AI 모델을 **격리해서 안전하게 굴리려면** 컨테이너 + 오케스트레이터 필요.

#### Docker Engine 최신 ✅

**무엇**: 컨테이너 만들고 실행하는 기본 엔진. 표준 중의 표준.
**왜 필요**: 모델별 의존성이 다 다름 (Python 버전, CUDA 버전 등). 컨테이너로 격리 안 하면 충돌 지옥.
**비유**: 각 셰프에게 별도 주방 칸 배정.

#### k3s v1.35.5+k3s1 ✅

**무엇**: **가벼운 Kubernetes 배포판** (이전 가이드에서 설명한 그것).
**왜 필요**: 컨테이너 7~10개 (Qwen, 임베딩, 리랭커, ASR…) 를 자동으로 관리. 죽으면 부활, 부하 분산.
**왜 k3s?**: 정통 k8s 는 무거움. k3s 는 단일 바이너리 + 적은 메모리 → 사내 서버 1~몇 대 구성에 딱.
**비유**: 식당 매니저. 셰프 출퇴근/배치/대체 인력 다 관리.

#### Helm v3.21.0 ✅

**무엇**: Kubernetes 의 **앱 설치 도구** (=`apt install` 의 k8s 버전).
**왜 필요**: 각 모델 띄울 때 YAML 10~20개 작성하는 노가다 → Helm 으로 한 줄.
**예시**: `helm install qwen-llm qwen/vllm --set model=Qwen/Qwen3-35B-A3B`
**비유**: 셰프 한 명 고용할 때 인터뷰부터 계약서까지 자동화한 절차.

#### NVIDIA Device Plugin 최신 ✅

**무엇**: k8s 가 **GPU 를 자원으로 인식**하게 하는 부품.
**왜 필요**: 기본 k8s 는 CPU/RAM 만 봄 → 이 플러그인이 "GPU 8장 있어요" 알려줌 → 모델별 GPU 배정 가능.
**비유**: 매니저에게 "가스레인지 8구 있다" 알려주는 신고서.

→ 이 4개가 컨테이너 스택 한 세트.

### 1-4. DB

#### PostgreSQL 16+ 📋 HFAgent-LAN

**무엇**: 오픈소스 관계형 DB. 가장 인기 있는 SQL DB.
**왜 필요**: 사용자/대화/메타데이터 저장. 신뢰성 + SQL 기능 풍부.
**왜 16?**: 성능 향상 + JSON 지원 강화 + 16/17 신규 기능.
**비유**: 식당의 매출/주문/재고 장부.
**📋 HFAgent-LAN**: 사내망 (LAN) 에 설치 예정. 외부 차단.

#### pgvector 📋 HFAgent-LAN

**무엇**: PostgreSQL **확장팩**. 벡터(임베딩) 저장 + 유사도 검색 기능 추가.
**왜 필요**: 별도 벡터DB (Qdrant 등) 안 깔고 PostgreSQL 한 곳에서 다 처리 가능.
**비유**: 장부에 "그림 검색" 기능을 추가하는 부록.
**언제 씀**: RAG 시 문서 검색 — "이 질문과 의미가 비슷한 문서 찾아줘".

→ PostgreSQL + pgvector 조합이면 일반 데이터 + 벡터 검색을 **한 DB 에서**.

## 2. AI 서비스 모델

### 2-0. 서빙 전략 — 2단계로 나눠 깐다

> 모델을 한 번에 다 열지 않고 **1차 → 2차** 순서로 단계적으로 서빙함.

```
[1차 단계]                          [2차 단계]
직원 → vLLM (포트 8100)      →      직원 → API Gateway (하니 9000 포트)
       └ 모델 1번만 직접 노출              └ 안에서 여러 모델로 분배
                                              (LLM/임베딩/리랭커/ASR/VL …)
```

#### 1차 — vLLM 단독 서빙 (포트 8100)

**무엇**: 우선 메인 LLM (1번 모델, Qwen3.6-35B) 만 vLLM 으로 띄워 **포트 8100** 에 직접 제공.
**왜**: API Gateway 가 아직 없으니, 모델 하나라도 먼저 열어 검증/사용 시작.
**임시성**: API Gateway 가 완성되면 **이 8100 직접 노출은 제거**됨 (Gateway 뒤로 숨김).
**비유**: 식당 오픈 전, 대표 메뉴 하나만 먼저 시범 판매.

#### 2차 — API Gateway 로 래핑 (하니 9000 포트)

**무엇**: vLLM 들을 감싸는 **API Gateway** 를 개발 → 직원은 **하니(HFAgent) 9000 포트** 하나로만 접속.
**왜 필요**:
- 입구를 하나로 통일 (8100, 임베딩, 리랭커… 흩어진 포트를 9000 하나로)
- 인증/로깅/요청 분배(라우팅)/속도제한 을 Gateway 가 일괄 담당
- 뒤쪽 모델을 자유롭게 추가/교체해도 직원이 쓰는 주소(9000)는 그대로

**확장 계획**: 9000 포트 뒤로 **다른 모델들도 추가 서빙** (임베딩·리랭커·ASR·VL·Gemma 등).
**비유**: 식당 정식 오픈 — 모든 주문을 한 창구(9000)에서 받고, 주방(여러 모델)으로 알아서 분배.

> 정리: **8100 = 임시 직통(1번 모델만) → 9000 = 정식 통합 창구(여러 모델)**. Gateway 완성 시 8100 직통은 닫힘.

---

### 2-1. 1차 서빙 모델 (vLLM 기반)

> 모두 Qwen + Gemma + pyannote 의 조합. 8장 GPU 에 분산.

#### 1️⃣ LLM — Qwen3.6-35B-A3B (GPU 1~4, 포트 8100) ⏳

**무엇**: 대화·글쓰기·추론하는 메인 LLM.
**스펙**: 35B 총 파라미터 / **3B 활성** (MoE, Mixture of Experts).
**MoE 란?**: 전문가 35명 회사인데 질문마다 가장 잘 아는 3명만 일함 → 빠름.
**GPU 4장 쓰는 이유**: 35B 는 1장 (80GB) 에 안 들어감 → 나눠 올림.
**용도**:
- 사용자 채팅
- 코드 생성
- 문서 요약
- RAG 응답 생성

**Claude 와 비교**: 일반 대화 90% 수준, 복잡 코딩/에이전트 60% 수준.

#### 2️⃣ 임베딩 — Qwen3-Embedding-8B (GPU 5) ⏳

**무엇**: 텍스트 → 숫자 벡터 변환.
**왜 필요**: "의미가 비슷한 문서" 를 찾으려면 텍스트를 좌표로 바꿔야 함.
**예시**:
```
"고양이는 귀엽다" → [0.23, -0.41, 0.88, ...]
"강아지는 사랑스럽다" → [0.25, -0.39, 0.91, ...]  ← 가까움
"오늘 점심 뭐 먹지" → [-0.78, 0.12, -0.33, ...]  ← 멀음
```
**용도**: RAG 1단계 (넓게 검색), 추천, 중복 탐지.

#### 3️⃣ 리랭커 — Qwen3-Reranker-8B (GPU 5) ⏳

**무엇**: 임베딩이 찾은 후보를 **다시 정밀하게 줄세움**.
**왜 필요**: 임베딩만으론 정확도 부족 → 리랭커가 "질문 vs 후보문서" 짝을 보고 점수.
**흐름**:
```
질문 → 임베딩 → 100개 후보 → 리랭커 → 상위 5개
       (넓게)              (정밀)
```
**왜 같은 GPU?**: 둘 다 8B 라 한 장에 같이 올라감. 호출 시점 다르니 충돌 X.

#### 4️⃣ ASR — Qwen3-ASR-1.7B (GPU 7) ⏳

**무엇**: **음성 → 텍스트** (Whisper 같은 거).
**스펙**: 1.7B 라 작고 빠름.
**용도**:
- 회의록 자동화
- 콜센터 녹취 분석
- 음성 명령

#### 5️⃣ VL — Qwen2.5-VL-7B-Instruct (GPU 7) ⏳

**무엇**: **이미지 + 텍스트 멀티모달**.
**용도**:
- 사진 설명
- OCR (글자 인식)
- PPT/스크린샷 분석
- 영수증 항목 추출

#### 6️⃣ LLM — google/gemma-4-26B-A4B-it (GPU 0) ⏳

**무엇**: Google 의 LLM. **백업/대체용 메인 모델**.
**왜 두 개?**:
- 모델별 강점 다름 (코드, 한국어, 영어, 추론)
- A/B 비교용
- 한 모델 장애 시 대체

**A4B-it**: 26B 총 / 4B 활성 / Instruction-tuned (대화용).

#### 7️⃣ 화자분리 — pyannote/speaker-diarization-3.1 (GPU 7) ⏳

**무엇**: 오디오에서 **누가 언제 말했는지** 구분.
**ASR 과 차이**:
- ASR: "뭐라고 말했는지"
- 화자분리: "누가 말했는지"

**합치면**: "0~5초 화자A: 안녕하세요 / 5~12초 화자B: 오늘은…"
**용도**: 회의록, 콜센터 분석.

---

### 2-2. 별도 추가 설치 예정

#### OCR — PaddleOCR v4 (CPU) ⏳

**무엇**: 중국 Baidu 의 **OCR 라이브러리**. 무료, 한국어 지원.
**왜 CPU?**: OCR 은 GPU 없이도 빠름. GPU 자원 아끼려 CPU 에.
**용도**:
- 스캔 문서 → 텍스트
- 영수증/명함 자동 인식
- VL 모델보다 빠르고 정확 (단순 글자 인식 한정)

**VL 과 차이**:
- PaddleOCR = 글자만 빠르고 정확
- VL = 글자 + 이미지 의미 이해 (느리지만 풍부)

#### 정렬 — Qwen3-ForcedAligner (GPU 7) ⏳

**무엇**: 음성 + 대본 → **단어 단위 정확한 타임스탬프**.
**ASR 과 차이**:
- ASR: "받아쓰기" (대본 모름)
- ForcedAligner: "대본 있는데 각 단어 몇 초에 나왔는지"

**용도**:
- 노래방 자막 정렬
- 립싱크
- 정밀 자막 (단어별 타이밍)
- TTS 학습 데이터 만들기

## 4. 자주 하는 질문

### Q1. 왜 도커가 아니라 k3s 까지 가?

도커만으로도 모델 1~2개는 OK. 하지만 모델 7~10개 + 자동 재시작 + GPU 분배 + 무중단 업데이트 까지 가려면 오케스트레이션 (k3s) 필수. 손으로 docker 관리는 운영 불가능.

### Q2. 왜 PostgreSQL 만 쓰고 Qdrant 같은 전용 벡터DB 안 써?

pgvector 가 충분히 빨라짐 (1M 벡터까진 무난). 두 DB 운영 부담 줄이는 게 더 큼. 대용량 가면 그때 Qdrant 검토.

### Q3. 왜 Qwen 만 잔뜩 쓰고 Gemma 도 같이?

Qwen 만 쓰면 단일 모델 의존. Gemma 가 다른 회사 모델이라 *교차 검증* 용. 또한 한국어/영어 강점이 모델마다 달라 라우팅으로 적재적소.

### Q4. CUDA 13 이면 PyTorch 호환되나?

PyTorch 2.5+ 부터 CUDA 12.x 지원. CUDA 13 은 매우 최신이라 일부 라이브러리 호환 작업 필요. vLLM 0.21 이 CUDA 13 대응 버전.

### Q5. GPU 8장이 항상 다 쓰임?

아니. 트래픽 따라 다름. Grafana 에서 GPU 사용률 보면서 모델별 분배 조정. 1~4 (Qwen LLM) 가 보통 가장 바쁨.

### Q6. 모델 업그레이드는 어떻게?

Helm 으로 `helm upgrade qwen-llm qwen/vllm --set model=새모델` 한 줄. k3s 가 무중단 롤링 업데이트. 문제 시 `helm rollback` 한 줄로 복원.

### Q7. 보안 인증서 만료되면?

Certbot 이 자동 갱신 (cron 또는 systemd timer). 90일 주기. 알람만 Grafana 에 설정.

### Q8. 왜 처음부터 9000 포트 하나로 안 하고 8100 부터?

API Gateway 개발에는 시간이 걸림. 그동안 메인 LLM 이라도 먼저 써야 하니 vLLM 을 8100 에 직통으로 열어둔 것. Gateway 가 완성되면 8100 직통은 닫고, 모든 모델을 9000 한 창구로 통일함. **"먼저 쓰고, 나중에 정리"** 전략.
